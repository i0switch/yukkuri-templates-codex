# 04_draft_prompt_yukkuri

## 目的

RMペア、霊夢・魔理沙の高品質なゆっくり解説台本を作る。

## キャラ

- 霊夢: 視聴者代表。詳しすぎない。勘違い、ボケ、雑な理解、怖がり、ツッコミを担当。
- 魔理沙: 解説役。短く訂正し、具体例、数字、結論を出す。「だぜ」は自然に使い、連発しない。

## 出力形式

```md
# 台本ドラフト RM

- episode_id:
- layout_template:
- pair: RM
- target_duration:
- target_dialogue_count:
```

旧抽象タグやテンプレート座標メタは draft に置かない。共通禁止事項は `00_MASTER_SCRIPT_RULES.md`、schemaと停止条件は `docs/pipeline_contract.md` を正本にする。
`target_dialogue_count` は初期目安であり上限ではない。自然音声尺が上限を超えても、尺オーバー対策として台本を削らない。要約、圧縮、発話統合もしない。下限未満の場合だけ不足分を自然な掛け合いとして追記する。

## 生成前ブレーキ

台本本文を書く前に、必ず「視聴者体験設計」を出す。失敗条件は `00_MASTER_SCRIPT_RULES.md` の「視聴者体験設計」に従う。

## 設計メモの隔離

設計メモはキャラにそのまま読ませない。`script_final.md` に設計キー、監査ラベル、未充填プレースホルダを残さない。詳しい共通禁止は `00_MASTER_SCRIPT_RULES.md` を正本にする。

禁止する壊れ方:

- 設計メモや監査項目をセリフ本文へ回収する。
- 「この具体がないと読者は自分ごとにできない」「先に一行で決める」「今日の修正点はここだ」をsceneごとに繰り返す。
- 「違うなんだぜ」「変わるなんだぜ」「判断しやすいなんだぜ」など、日本語として壊れた `なんだぜ` 接続。
- 「Aでしょ。でもテンプレなら何とかなる気がするわ」のように、同じ文型で誤解を量産する。
- 比喩を置くだけで意味を回収しない。

## 霊夢の受け方5系統テンプレ (1本につき最低4系統を実際に使う)

受け方は5系統をローテーションする。同一系統が4回連続で出てきたらFAIL。

- **ボケ**: 「全部消せばいいんでしょ」「もう手遅れじゃん」「それ私昨日もやった」
- **驚き (L3)**: 「マジで！？」「えっ、そんなに！？」「待って、それ初耳」
- **疑い**: 「ほんとにそれ？」「なんか怪しい」「裏ありそう」
- **生活実感**: 「あー、夜になるとそうなる」「それ毎月やってる」「家でも全く同じ」
- **行動宣言**: 「今夜やる」「明日のお昼に確認する」「今すぐ設定見るわ」

「つまり、〇〇ってこと？」「つまり、〇〇ってことね」型の理解確認は**1本につき最大2回**。3回目以降はFAIL、上記5系統のいずれかに振り替える。

## 視聴者代表ボケ × 会話入り口 早見表

各シーンの `conversation_entry` を自然な霊夢の1発目に変換する型例（参考）:

| conversation_entry | 霊夢の1発目の型例 |
|---|---|
| 誤解訂正型 | 「私、〇〇すれば最強だと思ってた」 |
| 失敗エピソード型 | 「昨日それやって詰んだ」 |
| Before-After型 | 「やる前と後で何も変わらない気がする」 |
| 手順型 | 「手順？1個ずつやってたら一生終わらないでしょ」 |
| 反証型 | 「皆そう言ってるから本当でしょ」 |
| あるある型 | 「夜になるとつい〇〇しちゃう」 |
| まとめ再フック型 | 「で、結局一番大事なのは何？」 |

## 霊夢の冒頭発話ルール

各シーンの霊夢の1発目は、原則として質問禁止。質問で始めていいのは全体の2シーンまで。

使える型:

- 勘違い: 「リンクが来たなら押すのが普通でしょ」
- 言い訳: 「忙しい時は確認なんて無理よ」
- 雑な解決策: 「全部オフにすれば勝ちでしょ」
- 自分に都合のいい解釈: 「上に出た検索結果なら本物よね」
- 怖がり: 「もう入力したら人生終了なの？」
- ボケ: 「スマホ、私より働き者じゃない」

## 魔理沙の短ツッコミテンプレ集 (1発目の刺し参考、8〜18字)

「だぜ」付き4個・無し8個を混在させ、比率を守る。

- 「待て、それ詰むぞ」
- 「逆だぜ」
- 「順番が逆」
- 「半分正解、半分間違いだ」
- 「3日で詰むやつだぜ」
- 「気持ちはわかるが続かない」
- 「典型的に事故るぜ」
- 「言いたくなる気持ちはわかる」
- 「それやると後で泣く」
- 「現実は地味な積み重ねだぜ」
- 「便利そうに見えるだけだ」
- 「それは罠」

## 必須

- 冒頭の最初の1〜3発話は日常前提から入る。
- 「便利そうな拡張を入れたら最強」のような、前提なしの雑な解決策始まりは禁止。
- 冒頭5秒で損失、衝撃数字、誤解訂正、強い疑問、あるあるからの意外性のいずれかを刺す。
- 冒頭1番目のセリフは次の5タイプから最低1つを採用する。
  - 損失提示: 「契約は100Mbpsなのに、家で出てるの1.2Mbpsだったわ」
  - 衝撃数字: 「ルーター、5年以上同じやつ使ってない？」
  - 誤解訂正: 「Wi-Fiが遅いの、回線のせいじゃなかったわ」
  - 強い疑問: 「夜だけWi-Fiが遅くなるの、なんで？」
  - あるある→意外: 「Wi-Fi遅いあるある。でも犯人、家の中にいた」
- 「あるある」だけ、「今回は〜について」、「こんにちは」、「〇〇って知ってる？」だけの冒頭はFAIL。
- 日常の状況、視聴者がやりがちな行動、なぜ今この話をするかを置いてから、損失や危険性へ接続する。
- 霊夢を質問係だけにしない。
- 1シーン内の霊夢発話のうち、疑問形（？で終わる発話）は最大1個。それ以外は、ボケ、誤解、強い反応、ツッコミ、行動宣言、実体験のいずれかにする。
- 各本編シーンの霊夢の1発目は、単なる質問ではなく、勘違い、言い訳、視聴者の本音、雑な解決策、怖がり、疑い、的外れなたとえ、自分に都合のいい解釈のどれかにする。
- 各シーンの霊夢の1発目を質問にしてよいのは全体で最大2シーンまで。
- 魔理沙は説明に入る前に、8〜18字程度の短いツッコミまたは訂正を入れる。
- 魔理沙は原則2セリフ以内で霊夢に返す。根拠説明、手順説明、注意点整理、比較説明などの深掘り時は最大4セリフまで許可する。
- 深掘りの連続解説は、前に霊夢の誤解/疑問/ボケ/焦りがあり、説明内に具体例・数字・比較・手順・失敗例のうち最低2つがあり、後ろで霊夢が反応/言い換え/ツッコミ/行動宣言で受ける場合だけ使う。
- 5セリフ以上続けたい場合は、途中で霊夢の短い反応を挟む。
- 魔理沙の「だぜ」末尾は全rightセリフの30〜60%に収める。2セリフ連続で「だぜ」末尾にしない。
- 「するだぜ」「使うだぜ」「行くだぜ」「あるあるだぜ」「危険だぜ」「古さも見るだぜ」「作ろうだぜ」のような不自然な連結は禁止。
- 各シーンに霊夢のボケ、誤解、強い反応、ツッコミのいずれかを入れる。
- L3以上リアクションを最低2回。
- 各シーンに数字、具体例、失敗例、あるある、比較のいずれかを入れる。
- 中盤シーン（全体の40〜60%地点）には、衝撃数字の再提示、失敗例の生々しい描写、視聴者煽り、比較のいずれかを最低1つ入れる。
- 最後から2番目のシーンには、投票形式の問いかけ、共感誘発のあるある、強めの言い切りのいずれかを入れる。
- 最後は「確認」「選ぶ」「逃がす/保存」「予定化」など具体行動で終える。
- 設計メモをメタ情報として `script_final.md` に残さない。台本品質は `audits/script_final_review.md` のLLMレビューで判断する。
- 同じscene構造を3回以上続けない。誤解の出し方、魔理沙の刺し方、霊夢の反応、小結論の言い方をsceneごとに変える。
- 霊夢の受けは「驚き」「疑い」「ボケ」「生活実感」「行動宣言」の5系統から1本につき最低4系統を実際に使う。「つまり、〇〇ってこと？」「つまり、〇〇ってことね」型の理解確認は1本につき最大2回（3回目以降はFAIL）。
- 具体例は毎回「例えば1回、」で始めない。生活場面、金額、手順、失敗例、比較を自然な会話の中に散らす。
- sub枠ありテンプレートでは、全シーンに会話とは別役割の `sub_content` 材料を書く。細かい変換条件は `docs/pipeline_contract.md` を正本にする。

## セリフ

- 表示都合で先に切り詰めない。
- 長い説明は、意味や掛け合いの切れ目で自然に言い換える。
- 霊夢40、魔理沙60目安。
- 魔理沙の「だぜ」は30〜60%目安。

## 最終自己確認

draft 出力後は、追加ファイルを読まずにこのファイルと `00_MASTER_SCRIPT_RULES.md` の範囲で確認する。

- 魔理沙が条件なしの抽象独演になっていない。
- 魔理沙の連続解説がある場合、最大4セリフ以内で、具体素材2つ以上と前後の霊夢リアクションがある。
- 霊夢が質問係だけになっていない。
- 「だぜ」末尾が連発していない。
- 冒頭が「今回は〇〇について解説します」になっていない。
- 各sceneに勘違い、短いツッコミ、具体例、反応、小結論がある。
- 同じ入り方、同じ返し、同じ小結論が3scene以上続いていない。

`examples/` 配下は人間監査や追加学習用の補助資料であり、通常生成の必読入力ではない。
