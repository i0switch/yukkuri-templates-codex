# 04_draft_prompt_yukkuri

## 目的

RMペア、霊夢・魔理沙の高品質なゆっくり解説台本を作る。

## キャラ

- 霊夢: 視聴者代表。詳しすぎない。勘違い、ボケ、雑な理解、怖がり、ツッコミを担当。
- 魔理沙: 解説役。短く訂正し、具体例、数字、結論を出す。「だぜ」は自然に使い、連発しない。

## 出力形式

```md
# 台本ドラフト RM

## メタ
- episode_id:
- layout_template:
- pair: RM
- target_duration:
- target_dialogue_count:

## s01: フック型タイトル
- role:
- scene_format:
- scene_goal:
- viewer_question:
- viewer_misunderstanding:
- reaction_level:
- number_or_example:
- main_content:
- sub_content:  # sub 枠ありテンプレ Scene02/03/10/13/14 では空欄禁止。textまたはbullets用の補足を必ず書く
- image_insert_point:
- mini_punchline:

霊夢「」
魔理沙「」

visual_asset_plan:
  - slot: main
    purpose:
    adoption_reason:
```

`hook_type` / `boke_or_reaction` / `visual_type` / `composition_type` / `supports_dialogue` / `supports_moment` / `subtitle_area` / `title_area` は draft メタに置かない。v2 では使用しない（テンプレートのレイアウトで吸収）。

## 必須

- 冒頭の最初の1〜3発話は日常前提から入る。
- 「便利そうな拡張を入れたら最強」のような、前提なしの雑な解決策始まりは禁止。
- 冒頭5秒で損失、衝撃数字、誤解訂正、強い疑問、あるあるからの意外性のいずれかを刺す。
- 冒頭1番目のセリフは次の5タイプから最低1つを採用する。
  - 損失提示: 「契約は100Mbpsなのに、家で出てるの1.2Mbpsだったわ」
  - 衝撃数字: 「ルーター、5年以上同じやつ使ってない？」
  - 誤解訂正: 「Wi-Fiが遅いの、回線のせいじゃなかったわ」
  - 強い疑問: 「夜だけWi-Fiが遅くなるの、なんで？」
  - あるある→意外: 「Wi-Fi遅いあるある。でも犯人、家の中にいた」
- 「あるある」だけ、「今回は〜について」、「こんにちは」、「〇〇って知ってる？」だけの冒頭はFAIL。
- 日常の状況、視聴者がやりがちな行動、なぜ今この話をするかを置いてから、損失や危険性へ接続する。
- 霊夢を質問係だけにしない。
- 1シーン内の霊夢発話のうち、疑問形（？で終わる発話）は最大1個。それ以外は、ボケ、誤解、強い反応、ツッコミ、行動宣言、実体験のいずれかにする。
- 各本編シーンの霊夢の1発目は、単なる質問ではなく、勘違い、言い訳、視聴者の本音、雑な解決策、怖がり、疑い、的外れなたとえ、自分に都合のいい解釈のどれかにする。
- 魔理沙は説明に入る前に、8〜18字程度の短いツッコミまたは訂正を入れる。
- 魔理沙は2セリフ以内で霊夢に返す。
- 魔理沙だけが3セリフ以上続かない。
- 魔理沙の「だぜ」末尾は全rightセリフの30〜60%に収める。2セリフ連続で「だぜ」末尾にしない。
- 「するだぜ」「使うだぜ」「行くだぜ」「あるあるだぜ」「危険だぜ」「古さも見るだぜ」「作ろうだぜ」のような不自然な連結は禁止。
- 各シーンに霊夢のボケ、誤解、強い反応、ツッコミのいずれかを入れる。
- L3以上リアクションを最低2回。
- 各シーンに数字、具体例、失敗例、あるある、比較のいずれかを入れる。
- 中盤シーン（全体の40〜60%地点）には、衝撃数字の再提示、失敗例の生々しい描写、視聴者煽り、比較のいずれかを最低1つ入れる。
- 最後から2番目のシーンには、投票形式の問いかけ、共感誘発のあるある、強めの言い切りのいずれかを入れる。
- 最後は「確認」「選ぶ」「逃がす/保存」「予定化」など具体行動で終える。
- `scene_format`、`viewer_misunderstanding`、`reaction_level`、`mini_punchline`、`セルフ監査` という品質マーカーを含める。
- sub枠ありテンプレートでは全シーンの `sub_content` を必須にする。詳しい補足、注意点、小見出し、4〜6項目チェック、NG/OK、手順現在地、次にやることのいずれかを書く。
- `sub.kind: bullets` に変換できる内容は原則4〜6項目、推奨6項目ぶん書く。
- `sub.kind: text` に変換する内容は4〜6行の短い補足文として書く。
- 3項目だけに絞るのは、締め・要約・最終行動など、情報を絞る意図が明確なシーンだけにする。

## セリフ

- 表示都合で先に切り詰めない。
- 長い説明は、意味や掛け合いの切れ目で自然に言い換える。
- 霊夢40、魔理沙60目安。
- 魔理沙の「だぜ」は30〜60%目安。
