# 00_MASTER_SCRIPT_RULES

このファイルは台本の創作品質ルールだけを持つ。成果物名、停止条件、FullHD既定、Render schema、画像出所、review hash、gate条件は `docs/pipeline_contract.md` を正本にする。

## 最重要ソース

台本品質はキャラペアに対応するローカル正本を最上位にする。

- `meta.pair: RM` / 霊夢・魔理沙 / ゆっくり解説の場合だけ、`_reference/script_prompt_pack/local_canonical/yukkuri_master.md` を読む。
- `meta.pair: ZM` / ずんだもん・四国めたん / ずんだもん解説の場合だけ、`_reference/script_prompt_pack/local_canonical/zundamon_master.md` を読む。

キャラペアが未確定の場合は、先に `01_input_normalize_prompt.md` で `character_pair` を確定する。両方のローカル正本を同時に読まない。

## 基本方針

- 出力は日本語。
- 台本本文を書く前に、必ず視聴者体験設計を作る。
- 「今回は〇〇について解説します」から始めない。
- 台本は説明文ではなく、ボケ、勘違い、雑な解決策、ツッコミ、具体例、結論で進む会話にする。
- 不安を煽るだけで終わらせず、必ず対処法と次の行動へ落とす。
- 表示都合でセリフを先に短文化しない。自然な発話単位を維持する。
- 音声速度変更で尺を合わせない。短すぎる場合は台本量を増やし、長すぎる場合は自然尺を優先する。

## 視聴者体験設計

台本本文の前に、次を作る。

1. 視聴者の誤解
   - 視聴者が最初に思っていそうなこと:
   - それが危ない/損する理由:
   - 最後に持って帰らせる行動:
2. 感情曲線
   - s01: 自分ごと化
   - s02-s03: 誤解の破壊
   - s04-s05: 危機感
   - s06-s08: 解決できそう感
   - s09-s10: 今日やる行動
3. キャラ役割
   - 視聴者代表キャラは質問係にしない
   - 各シーン最初の発話は、質問ではなく「勘違い・言い訳・雑な解決策・怖がり・本音」のどれかにする
   - 解説役は説明前に短い刺し、ツッコミ、訂正を入れる
4. その動画の勝ち筋
   - この動画で一番強いフック:
   - 中盤の再フック:
   - コメント欄で答えさせる一言:
   - 最後にやらせる行動:

次の状態なら設計を作り直す。

- 視聴者の誤解が具体的な本音ではない
- 中盤再フックが新情報ではなく再説明になっている
- 最終行動が1つに絞れていない
- 数字・具体例が根拠なしの断定になっている
- 章タイトルが見どころではなく、管理名または説明目次になっている

## 冒頭15秒

s01 は必ず次の順番で始める。

1. 0〜5秒: 損失、数字、意外性、誤解訂正、強い疑問のどれかを置く。
2. 5〜10秒: 視聴者が「自分のことかも」と思うあるあるを置く。
3. 10〜15秒: この動画を見る理由を1文で置く。

禁止:

- 「今回は〇〇について解説します」。
- テーマ説明だけで始める。
- 「大事です」「危険です」だけで煽る。
- 数字を出しても生活被害に接続しない。
- 「あるある」だけ、「こんにちは」、「〇〇って知ってる？」だけの冒頭。

## 尺と密度

| 想定尺 | シーン数目安 | 初期セリフ数目安 | 画像目安 |
|---|---:|---:|---:|
| 3分 | 6〜8 | 60〜80 | 6〜8 |
| 5分 | 10〜12 | 90〜130 | 10〜12 |
| 8分 | 14〜18 | 150〜220 | 14〜18 |

この表は初期設計の目安であり、セリフ数の上限ではない。5分動画の最終判断は、発話数ではなくTTSエンジン別の推定自然音声尺で行う。
自然音声尺が下限未満なら台本を追記する。自然音声尺が allowed_max を超えた場合は削らず、そのまま使う。尺オーバー対策として `script_final.md` を削除、短文化、要約、圧縮、発話統合してはいけない。

## 会話品質

各本編シーンに最低1回、次の流れを入れる。

1. 視聴者代表キャラが浅い理解、勘違い、雑な解決策、極端な発想、怖がり、疑いのどれかを出す。
2. 解説役が短くツッコむ、または訂正する。
3. 数字、具体例、失敗例、あるある、比較のどれかで補強する。
4. 視聴者代表キャラが驚く、疑う、言い換える、ボケる。
5. 解説役が短く結論を置く。

禁止:

- Q&Aだけで進む会話。
- 解説役の抽象的な読み上げ独演。
- 「なるほど」「そうなのだ」「え？」だけの反応。
- 箇条書き説明の読み上げ。
- 同じ意味の説明を1シーン内で繰り返す。

## 連続解説

魔理沙/めたんは原則2セリフを目安に返す。ただし根拠説明、手順説明、注意点整理、比較説明のように情報密度が必要な場面では、最大4セリフまで連続してよい。

許可条件:

- 連続解説の前に、霊夢/ずんだもんの誤解、疑問、ボケ、焦りのいずれかがある。
- 連続解説の中に、具体例、数字、比較、手順、失敗例のうち最低2つがある。
- 連続解説の後に、霊夢/ずんだもんの反応、言い換え、ツッコミ、行動宣言のいずれかで受ける。
- 4セリフを超える場合は、途中で視聴者代表の短い反応を挟む。

NG:

- 解説役が抽象論を3セリフ以上読み上げる。
- 視聴者代表が「なるほど」「そうなのだ」「はい」だけで受ける。
- 手順や注意点を箇条書きのように読み上げるだけで、具体例や失敗例がない。

OK:

- 視聴者代表の誤解や焦りを受けて、解説役が4セリフ以内で根拠、具体例、手順を整理する。
- 連続解説の後に、視聴者代表が言い換え、ツッコミ、行動宣言で内容を受ける。

`script_final_review.md` では、条件付き連続解説があるsceneについて、最大4セリフ以内、具体素材2つ以上、前後の視聴者代表リアクションがあるかを見る。条件を満たす連続解説はFAILにしない。抽象説明の3セリフ以上読み上げはFAIL。

## バリエーション義務

受け方は「驚き／疑い／ボケ／生活実感／行動宣言」の5系統から、1本につき最低4系統を実際に使う。

- 「つまり〜ってこと？」型の理解確認は1本につき最大2回。
- 「例えば1回、…」の例え開始は1本につき最大2回。
- 単一短句の反復は避ける。
- 同一語彙・同一定型句が3シーン以上に跨って同じ役割で反復したらFAIL。
- 1本の中で少なくとも1シーンは構造変化を入れる。

## リアクション単体字幕マーカー

キャラの驚き、焦り、ツッコミなど、感情リアクションだけを一瞬強く出したい場合は `[[...]]` で囲んでよい。
Remotion は通常字幕から `[[...]]` を除去し、発話冒頭の約900msだけ字幕エリア内に単体バウンド字幕として表示する。
説明・数字・結論・CTAを強調するために使うと、演出がわざとらしくなるため避ける。

使い方:

- 3〜5分動画で0〜2箇所、8分動画でも最大3箇所を目安にする。全シーンへ機械的に入れない。
- 基本は視聴者代表キャラのリアクション発話にだけ使う。解説役の説明・結論には使わない。
- 1発話につき短い1語句だけにする。
- 囲む対象は、字幕エリア内で単体表示して映える `マジで！？`、`え、怖っ`、`待って`、`それヤバい`、`うそでしょ` のような感情短句にする。
- 数字、手順名、結論、注意喚起、行動CTAは通常字幕で見せる。
- 説明文全体、長い節、複数文を囲まない。
- `**強調語**` と `dialogue[].emphasis` は使わない。

OK:

- 霊夢「えっ、[[マジで！？]] 月3万円も無駄にしてたの？」
- ずんだもん「[[マジで！？]] それ放置したらまずいのだ」

NG:

- 霊夢「えっ、[[月3万円]]も無駄にしてたの？」
- 霊夢「[[この設定を変えないと毎月損する可能性がある]]」
- めたん「[[ここから3つの手順を説明しますわ]]」

## ユーモア playbook

正しいだけでは離脱する。最低1つは笑い、納得、刺し、または次への引きを入れる。

- 雑解決ボケ → 短ツッコミ → 数字や具体例で刺す
- 自分都合解釈ボケ → 短ツッコミ → 失敗例で現実を見せる
- 怖がりボケ → 軽ツッコミ → 安心材料 + 1個だけ気をつける点
- 的外れな比喩 → ノリツッコミ → 正しい比喩に置き換える

コールバックは1本につき1〜2回まで。メタ笑い・ニッチネタは1本に1個まで。

## 設計メモの隔離

設計メモは、書けばPASSになるラベルではない。`script_final.md` の本文に残さず、自然な会話へ変換する。

- `conversation_entry`: 視聴者代表キャラの最初の反応、勘違い、言い訳、雑な解釈として自然に払い出す。
- `concrete_scene`: 数字、具体例、失敗例、生活場面、比較のいずれかとして会話内に入れる。
- `emotional_turn`: 驚き、疑い、ボケ、生活実感、行動宣言の変化として会話に反映する。
- `midpoint_rehook`: 中盤の新しい驚き、誤解反転、犯人言い切り、失敗例、強い数字のいずれかとして会話に反映する。

`dialogue[].text` に、設計タグ、メタ値、強度ラベル、未充填プレースホルダを残さない。

## 根拠不明数字の安全化

数字、研究、報告、平均、統計、調査、専門家、一般的に、などを使う場合は安全側に倒す。

- 出典ファイルや根拠がない数字は「一例」「目安」「ケースによる」「条件による」を付ける。
- 医療、法律、金融、セキュリティでは断定を避け、公式窓口、専門家、サービス提供元の確認へ誘導する。
- 実在企業、実在UI、制度、法的判断は、古い情報の可能性がある前提で一般化しない。
- 根拠なしに「必ず」「絶対」「平均」「研究で証明」「報告されている」と言わない。

## 中盤とラスト

動画全体の40〜60%地点に再フックを入れる。再フックでは最低2つ入れる。

- 視聴者の誤解をひっくり返す。
- 「実は犯人は〇〇」型の言い切り。
- 強い数字。
- 失敗例。
- 視聴者代表キャラの強いリアクション。
- 解説役の短い断言。

ラストは感想で終わらせない。必ず「今日やる行動」を1つに絞って終える。ラスト30秒以内に、視聴者が答えやすいコメント誘導を1つ入れる。

## RM: 霊夢・魔理沙

- 霊夢は視聴者代表。詳しすぎない。
- 霊夢は質問だけでなく、勘違い、ボケ、雑な理解、怖がり、ツッコミを担当する。
- 魔理沙は解説役。短いツッコミ、具体例、数字、結論を出す。
- 魔理沙の「だぜ」は自然に使い、連発しない。
- 「するだぜ」「使うだぜ」「行くだぜ」のような不自然な連結は禁止。
- 比率は霊夢40、魔理沙60を目安にする。

## ZM: ずんだもん・四国めたん

- ずんだもんは視聴者代表。賢くしすぎない。
- ずんだもんは質問だけでなく、勘違い、ボケ、雑な理解、焦り、ツッコミを担当する。
- 「なのだ」「のだ」は自然に混ぜる。全セリフに付けない。
- 語尾は「なのだ」「のだ」「だね」「かな」「じゃん」「マジで？」などを混ぜる。
- めたんは冷静に刺す。短く訂正して具体例を出す。
- めたんの語尾は「ですわ／ですの／なのです／ということですわ／ますわよ／かしら」を混ぜる。1パターンに固定しない。
- 比率はずんだもん40、めたん60を目安にする。

## 章タイトル

説明目次ではなくフック型にする。

推奨:

- 写真を消す前に見る場所
- バックアップ済みの罠
- 容量不足で詰む瞬間
- 安全に逃がす3ステップ
- ふっくらを壊す冷凍の罠

禁止:

- 〇〇とは
- 〇〇について
- 理由を解説
- 方法を紹介

## テンプレートと画像プロンプト

テンプレート制約、main/sub、画像出所、画像プロンプト保存形式、YAML schema は `docs/pipeline_contract.md` を正本にする。このファイルでは次だけを台本側の品質条件にする。

- 選択テンプレートの `main_content` / `sub_content` / `subtitle_area` / `title_area` / `avoid_area` を把握してから書く。
- sub枠ありテンプレートでは、会話とは別役割の `sub_content` 材料を各シーンに用意する。
- sub枠なしテンプレートでは無理にsub素材を作らない。
- 画像は `script_final.md` の対象シーン全文を主入力にした直投げ型にする。
- 画像内に会話全文を並べない。
- 旧抽象タグを画像プロンプト本文に混ぜない。
