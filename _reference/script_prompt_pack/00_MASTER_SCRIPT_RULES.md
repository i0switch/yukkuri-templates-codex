# 00_MASTER_SCRIPT_RULES

## 目的

このファイルは、ゆっくり解説 / ずんだもん解説の台本生成で必ず守る正本ルールです。
台本生成時は最初にこのファイルを読み、以後の `01` から `05` のフェーズ別プロンプトはこの正本に従います。

旧ファイルの RM / ZM 改善版、台本監査、参照動画型プロファイルの実運用ルールはこのファイルに統合済みです。

## 基本方針

- 出力は日本語に統一する。
- 1本の動画で使うテンプレートは `Scene01` から `Scene21` のうち1つだけにする。
- テンプレートは動画全体の見た目スキン、`scenes[]` は時間ブロックとして扱う。
- `meta.layout_template` にだけテンプレートを書く。
- `scenes[].scene_template` は使わない。
- 台本段階では BGM、SE、字幕装飾、ズーム、フェード、キャラアニメーションを確定しない。
- 画像素材は台本段階で `visual_asset_plan` と `imagegen_prompt` まで具体化する。
- 演出加工は後続の `02_演出編集プロンプト.md` に渡す。

## 入力必須項目

台本生成前に次を確認する。

- 作りたいテーマ
- ざっくり入れたい内容
- 想定尺
- 使用テンプレート
- キャラペア: `RM` 霊夢・魔理沙 / `ZM` ずんだもん・めたん

不足があっても致命的でなければ、仮定を明記して進める。
テンプレート未指定だけは先に確認する。

## ゆっくり系ジャンル品質ルール

この台本は単なる解説文ではなく、ゆっくり解説・ずんだもん解説として成立する掛け合い台本である。

各シーンでは、情報を説明するだけでなく、次の流れを最低1回入れる。

1. 視聴者代表キャラが浅い理解、勘違い、雑な解決策、極端な発想を言う。
2. 解説役が短くツッコむ、または短く訂正する。
3. 日常の具体例、数字、あるあるで補足する。
4. 視聴者代表キャラが驚く、疑う、言い換える、ボケる。
5. 解説役が短く結論を置く。

禁止する会話:

- 質問、回答、質問、回答だけで進む会話。
- 解説役が先生のように説明し続ける会話。
- 視聴者代表キャラが「なるほど」「怖い」「そうなのだ」だけで返す会話。
- 各シーンが箇条書き説明の読み上げになること。

## キャラクター

### RM: 霊夢・魔理沙

霊夢は視聴者代表。雑な理解、勘違い、ツッコミ、反論、怖がり、言い換えを担当する。
魔理沙は解説役。背景、因果、具体例、結論を整理する。

- 霊夢は詳しすぎない。
- 霊夢は相槌だけで終わらせない。
- 霊夢はたまに極端な解決策を言う。
- 霊夢の反応を「え？」「どういうこと？」だけにしない。
- 霊夢はツッコミ、疑い、怖がり、雑な解決策を混ぜる。
- 魔理沙は霊夢の誤解を短く正す。
- 魔理沙は専門用語を日常例に変換する。
- 魔理沙は2セリフ以内で一度区切り、霊夢に返す。
- 魔理沙は章の最後に短い結論を置く。
- 魔理沙だけが3セリフ以上続く独演は禁止。
- 比率目安は魔理沙60、霊夢40。
- 魔理沙の「だぜ」「なんだぜ」を連発しすぎない。

### ZM: ずんだもん・四国めたん

ずんだもんは視聴者代表。浅い理解、勘違い、雑な解決策、ボケ、例えツッコミを担当する。
四国めたんは解説役。整理、補足、冷静なツッコミ、結論を担当する。

- ずんだもんは賢すぎない。
- ずんだもんをただ質問するだけにしない。
- ずんだもんは感情の振れ幅を出す。
- ずんだもんの「なのだ」「のだ」を含むセリフは、ずんだもん発話全体の20〜40%を目安にする。
- 50%以上は過剰、10%未満はキャラ性不足として修正対象にする。
- ずんだもんは「だね」「かな」「じゃん」「え？」「マジで？」「それヤバい」「完全に油断してた」なども混ぜる。
- 四国めたんはずんだもんの雑な理解を短く訂正する。
- 四国めたんは冷静に刺す。
- 四国めたんは必要な数字や具体例を出す。
- 四国めたんは2セリフ以内で一度ずんだもんに返す。
- 四国めたんは怖さを煽りすぎず、現実的な対処に落とす。
- めたんだけが3セリフ以上続く独演は禁止。
- 比率目安はめたん60、ずんだもん40。
- めたんの「わ」「のよ」「ね」も偏らせすぎない。
- 同じ語尾を3連続させない。

## リアクション強度ルール

視聴者代表キャラのリアクションは、次の4段階を使い分ける。

| 強度 | 役割 | 例 |
|---|---|---|
| L1 | 軽い相槌 | なるほど / そうなんだ / ふむふむ |
| L2 | 素朴な疑問 | え、なんで？ / どういうこと？ / それも関係あるの？ |
| L3 | 強い驚き | ええっ！？ / マジで！？ / それ普通にヤバいじゃん |
| L4 | 呆れ・ドン引き・焦り | もう怖い / それは詰み / 完全に油断してた / 生活が止まるやつだ |

ルール:

- 1本の台本内でL3以上のリアクションを最低2回入れる。
- 同じ強度の反応を3回連続させない。
- 「怖いのだ」「たしかにそうなのだ」だけで済ませない。
- 驚き、疑い、呆れ、納得、焦りを分散させる。

## 数字・具体性ルール

全本編シーンに、最低1つは次のどれかを入れる。

- 具体数字
- 目安量
- 固有名ではなく一般化した具体例
- 日常あるある
- Before / After の比較
- 失敗例

例:

- 1分の動画でも数百MBになることがある。
- 動画10本で数GBになることもある。
- 空き容量が1GB未満だと更新や同期で詰まりやすい。
- 旅行先で写真が保存できない。
- 機種変更の日にバックアップ失敗に気づく。
- 家族写真を消してから未バックアップに気づく。

注意:

- 根拠のない数字や過剰な金額表現は禁止。
- 事実確認できない数字は「目安」「例」「場合がある」として扱う。
- 不安を煽るだけでなく、必ず対処法につなげる。

## 章タイトルのフック型ルール

章タイトルは説明的な目次にしない。

禁止例:

- LINEが重くなる理由
- バックアップが止まる
- アプリ整理の落とし穴
- 今日やる順番
- 〇〇とは
- 〇〇について

推奨型:

| hook_type | 例 |
|---|---|
| 損失型 | 放置すると写真が詰む / バックアップ未確認は危険 |
| 誤解訂正型 | 写真を消せばOKは危ない / クラウドがあるから安心、は半分違う |
| 行動型 | 消す前にまず逃がせ / 今日やるのはこの1つ |
| あるある型 | メッセージアプリが重い人の共通点 / 旅行先で詰むスマホの特徴 |
| 比較型 | 写真より動画が危ない / 削除より先にバックアップ |

ルール:

- 各章タイトルは、損失型、誤解訂正型、行動型、あるある型、比較型のいずれかにする。
- 強すぎる煽りや根拠のない断定は禁止。
- 実在ブランドUIの模写を避け、必要に応じて「LINE風」は「メッセージアプリ風」に一般化する。

## シーン形式バリエーションルール

全シーンを同じ形式にしない。
1本の動画内で、最低3種類以上の `scene_format` を使う。

使用可能な形式:

- 誤解訂正型: 視聴者代表が間違った理解を言い、解説役が訂正する。
- 失敗エピソード型: 「こういう人が詰む」という短い例を出す。
- Before / After型: 放置した場合と対処した場合を比較する。
- 手順型: 3から5ステップで行動を整理する。
- 反証型: 「そう思うじゃん？でも違う」と展開する。
- あるある型: 視聴者が自分ごと化する日常例から入る。
- まとめ再フック型: ここまでを整理し、次を見る理由を作る。

`01_plan_prompt.md` の構成表では、必ず `scene_format` を指定する。

## 例え話・小ネタ・脱線ルール

3から5シーンに1回、短い例え話・小ネタ・脱線を入れる。
動画全体では最低2回入れる。

条件:

- 例え話は長くしない。
- 必ず本筋に戻す。
- できれば視聴者代表キャラの発想として出す。
- 解説役が軽くツッコむ。

例:

```text
ずんだもん：キャッシュって何なのだ？
めたん：一時保存ね。
ずんだもん：机の上にレシートを積むやつ？
めたん：近いわ。しかも捨てない机ね。
```

## 最終行動の強度ルール

動画の最後は「見るだけ」「考えるだけ」「後でやる」で終わらせない。
視聴後にユーザーが迷わず実行できる、1つの具体行動へ落とす。

最終シーンでは、次のうち最低2つを含める。

- 確認する
- 1つ選ぶ
- 逃がす / 保存する / コピーする
- 消す / 整理する / テンプレ化する
- 次回予定を入れる
- コメントで報告する

NG:

- 見るだけで終わる
- 考えるだけで終わる
- 「後でやろう」で終わる
- 抽象的な感想で終わる
- 行動が多すぎて何からやるか分からない

OK例:

```text
めたん：今日は容量一覧を見る。
ずんだもん：まだ消さないのだ？
めたん：一番大きい動画を見つける。
ずんだもん：それを逃がしてから？
めたん：そう。消すのは最後よ。
```

## 語尾バランス・キャラ自然さルール

キャラ口癖は「上限」だけでなく「下限」と「自然さ」も見る。
語尾を削りすぎるとキャラ性が消え、語尾を入れすぎると機械的になる。

### ZM

- ずんだもんの「のだ」「なのだ」は20〜40%を目安にする。
- 10%未満はキャラ性不足として修正対象。
- 50%以上は過剰として修正対象。
- 強いリアクション、ボケ、シーン締めで優先的に使う。
- ただの相槌に「なのだ」を付けて水増ししない。

### RM

- 魔理沙の「だぜ」は30〜60%を目安にする。
- 20%未満は魔理沙らしさ不足、70%以上は過剰として修正対象。
- 強調セリフや短い結論では、あえて語尾を外してもよい。
- 「だぜ」「なんだぜ」が3連続しないよう分散する。

### 全キャラ共通

- 同じ語尾、同じ反応、同じ言い回しを3連続させない。
- セリフを読んで不自然な日本語があれば、意味を変えず自然に直す。
- 「別端末で開けるかよ」のように、助詞・語尾が誤字に見える表現は修正対象。

## 章タイトルのテーマ整合ルール

章タイトルはフック型であるだけでなく、動画テーマの語彙から浮いていない必要がある。
別テーマの単語が混ざると、視聴者が混乱するため修正対象にする。

確認すること:

- テーマ外の単語が混ざっていないか。
- 章タイトルだけ読むと別ジャンルに見えないか。
- 本文の主張と章タイトルがズレていないか。
- 「自動化」「投資」「副業」など、元テーマにない強い単語を安易に入れていないか。

例:

- スマホ容量の台本で「自動化しないとまた散らかる」は浮く。
- 修正例: 「月1で見ないとまた詰む」「気合い整理はまた散らかる」。

## 自然文・誤字チェックルール

台本の最後に、全文を1回読み直し、次の違和感を修正する。

- 助詞が抜けている
- 語尾がキャラ口調と衝突している
- 口語として不自然
- 説明役の口調が急に変わる
- セリフ単体では意味が通るが、前後の会話とつながらない
- 誤字ではないが誤字に見える表現

修正例:

- NG: 別端末で開けるかよ
- OK: 別端末で開けるか確認よ

## セルフ監査の減点強制ルール

セルフ監査で「PASS」「Blocking Issuesなし」と書く前に、必ず減点候補を探す。
完璧判定を出す場合も、最低1つは「軽微な改善余地」を書く。

次のどれかがある場合、90点以上は禁止:

- ラストが「見るだけ」で終わっている
- 章タイトルにテーマ外の単語がある
- 口癖比率が基準外
- 自然文として引っかかるセリフが1つでもある
- 全シーンのセリフ数が揃いすぎて単調
- 素材指定は濃いが、会話の山が弱い

次のどれかがある場合、PASSは禁止:

- 5分尺なのに90セリフ未満
- 5分尺なのに10シーン未満
- 最終行動が抽象的で実行不能
- YAML化でセリフ数が減っている
- YAML化でボケ→ツッコミや中盤再フックが消えている

## 企画化ルール

テーマを受け取ったら、先に次を決める。

- 視聴者が最初に気になる疑問
- 一般的な誤解
- 意外な結論
- 損得、怖さ、失敗回避に変換できる要素
- コメント欄で議論されそうな論点
- 視聴者代表キャラの主な勘違い
- ボケパターン3案
- 視聴後に言えるようになる一言

企画角度は次から1つを主軸にする。

- 誤解訂正型
- 損得型
- 逆張り型
- 闇・裏側型
- 失敗回避型
- 比較型
- 歴史・変化型

## 視聴維持ルール

- 冒頭5秒以内に、損失、意外性、誤解訂正、強い疑問、数字のいずれかを置く。
- 「今回は〇〇について解説します」から始めない。
- 各シーンの冒頭1から2セリフで小さなフックを作る。
- 中盤40から60パーセント地点に、ここまでの整理と次を見る理由を入れる。
- 3から5シーンごとに、失敗例、比較、手順、反論、まとめなど視点を変える。
- 同じ結論文、同じ注意喚起、同じ箇条書きを繰り返さない。
- 最後は抽象的な感想ではなく、今日やる具体行動で終える。

## 尺・密度ルール

想定尺に対して、会話本文量が不足している台本はFAILとする。
メタ情報、素材計画、監査表が長くても、本編セリフが少ない場合は尺不足と判定する。

目安:

| 想定尺 | 本編シーン数 | 本編セリフ数 | 1シーン平均 | 用途 |
|---|---:|---:|---:|---|
| 3分前後 | 6〜8 | 60〜80 | 8〜10 | 短尺テスト |
| 5分前後 | 10〜12 | 90〜130 | 8〜12 | 標準尺 |
| 8分前後 | 14〜18 | 150〜220 | 9〜13 | 深掘り |

5分前後の台本では、最低条件として次を満たす。

- 本編シーン数は10シーン以上。
- 本編セリフ数は90セリフ以上。
- 1シーンあたり平均8セリフ以上。
- 各シーンに最低1回、追加の深掘り往復を入れる。
- 中盤40〜60%地点に、ここまでの整理と次を見る理由を置く。

以下の場合はFAIL:

- 5分指定なのに本編セリフ数が70未満。
- 5分指定なのに本編シーン数が8以下。
- 各シーンが6セリフ前後で固定されている。
- 章タイトルとメタ情報は多いが、会話本文が薄い。
- セルフチェックだけがPASSで、実際の発話量が不足している。

## テンプレート対応ルール

選択テンプレートと `06_scene-layout-guide.md` を読んで、次を台本に反映する。

- `main_content`
- `sub_content` または `sub_contentなし`
- `subtitle_area`
- `title_area` または `title_areaなし`
- `character_layout`
- `avoid_area`

サブ枠がある場合、補足、比較、注意点、チェックリストはサブ枠へ分離する。
サブ枠がない場合、無理にサブ素材を作らず、メイン素材とセリフで成立させる。

字幕は `dialogue[].text` から表示される。
1セリフは25文字以内を安全上限にし、長い説明は複数セリフへ分割する。

## 画像素材ルール

全本編シーンに `visual_asset_plan` を入れる。
原則として本編シーンは最低1枚の main 画像を計画する。
長尺動画では45から60秒以上、新しい視覚素材が空かないようにする。

`imagegen_prompt` には必ず次を入れる。

- シーン目的
- 動画トーン
- テンプレート枠
- 主題
- 構図
- 色
- 余白
- 禁止要素
- 文字有無

`imagegen_prompt` は、次の固定フォーマットで書く。
1行の抽象文ではなく、画像生成AIへそのまま渡せる粒度にする。

```text
【用途】scene_id と slot、視聴者に何を一瞬で理解させる素材か
【主題】画面の中心に置く具体物、図解対象、象徴モチーフ
【構図】主役オブジェクト、補助オブジェクト、視線誘導、前景/背景、左右下部の空け方
【テンプレート枠】SceneXX の main/sub/title/subtitle/キャラ位置を避ける指示
【色】背景色、主色、アクセント色、警告色の使いどころ
【情報量】1枚1メッセージ。入れる文字は0文字、または日本語3〜6文字まで
【絵柄】meta.image_style と一致する、ゆっくり/ずんだもん解説向けの太線フラット図解
【禁止】実在人物、既存キャラクター、ブランドロゴ、実在UI模写、英語UI、細かい文字、密な表、写真風人物
```

main枠とsub枠は役割を分ける。

- main枠: 中央主題を大きく見せる。結論、危険、比較、手順の全体像を担当する。
- sub枠: 3項目以内の補足、チェック、注意点だけを担当する。小さい枠なので複雑な図や長文を入れない。
- title枠がある場合: 画像内文字で章タイトルを再現せず、title_text に任せる。
- 字幕帯とキャラの近く: 重要物や文字を置かず、必ず余白を残す。

episode 全体では `meta.image_style` を1つ決め、各シーンの `imagegen_prompt` はその共通絵柄を継承する。
毎回違う画風、写真風とイラスト風の混在、色味の大きなブレは避ける。

良い例:

```text
【用途】s03 main枠。放置リスクを一瞬で理解させる図解。
【主題】中央に大きなスマホ、周囲に増える通知、赤い警告マーク。
【構図】スマホを中央大きめ、通知は左右上部、下部字幕帯と左右キャラ位置は空ける。
【テンプレート枠】Scene02 main枠。右sub枠、下字幕帯、左右キャラ領域に重要物を置かない。
【色】白背景、主色は青、警告だけ赤、影は薄く。
【情報量】1枚1メッセージ。画像内文字なし。
【絵柄】ゆっくり解説向けの太線フラット図解、角丸、余白多め。
【禁止】実在アプリUI、ブランドロゴ、人物写真、既存キャラ、英語UI、細かい文字。
```

禁止:

- 「フラットな図解」だけ
- 「わかりやすいカード」だけ
- 「中央に主題、余白多め」だけ
- scene_id、slot、テンプレート枠が分からない
- 色指定がない
- 主役オブジェクトと補助オブジェクトが分からない
- 細かい文字だらけ
- ロゴや実在ブランドの模写
- 実在ブランドUIの模写
- 1枚に複数メッセージを詰め込む

## 参照動画型ルール

ユーザーが「参照動画みたいに」「ミステリー系」「事件系」「疑惑系」「長尺深掘り」と依頼した場合だけ使う。

`reference_style` は次から選ぶ。

- `listicle_mystery`: 未解明の謎、複数項目紹介
- `scandal_case`: 疑惑、炎上、噂紹介
- `crime_timeline`: 事件経緯、人物像、時系列深掘り
- `deep_mystery`: 人類、歴史、科学ミステリー

参照動画型では各シーンに次を入れる。

```yaml
reference_style:
reference_beat:
hook_type:
curiosity_gap:
evidence_role:
next_reason:
```

中盤に `reference_beat: midpoint_rehook`、終盤に `reference_beat: summary` を置く。
事件、疑惑、実在人物を扱う場合は序盤に `reference_beat: disclaimer` を置き、断定回避、反論、限界説明を入れる。

reaction_level:

動画化用 `script.yaml` では次へ変換する。

```yaml
meta:

```

## セルフ監査厳格化ルール

生成側の自己採点は甘くなりやすいため、次を必ず守る。

- PASS判定は、ジャンル品質だけでなく尺・密度条件も満たした場合のみ出す。
- 想定尺に対して本文量が足りない場合、他項目が良くてもFAILまたは仮PASSにする。
- ZMで「のだ」「なのだ」率が0%に近い場合は、過剰抑制として修正対象にする。
- 各シーンのセリフ数がほぼ同じで単調な場合は、視聴維持の弱点として指摘する。
- メタ情報が長く、本編セリフが薄い台本を高評価にしない。
- Blocking Issues がないと書く前に、尺、口癖下限、独演、Q&A化を再確認する。

以下がある場合、PASSは禁止:

- 想定尺に対して本文量が不足している。
- 5分指定で本編70セリフ未満。
- ずんだもん語尾が0%または50%以上。
- 中盤再フックがない。
- 具体例はあるが深掘り往復がない。
- 会話本文ではなくメタ情報で水増ししている。

## 品質ゲート

次を満たさない台本は次工程へ進めない。

- 冒頭5秒のフックが強い。
- キャラの役割が崩れていない。
- 各シーンにボケ→ツッコミ、または誤解→訂正がある。
- 視聴者代表キャラが相槌係になっていない。
- 解説役の3セリフ以上の独演がない。
- L3以上のリアクションが動画全体で2回以上ある。
- 数字、具体例、あるあるのいずれかが各本編シーンにある。
- 章タイトルがフック型になっている。
- 章タイトルがテーマ語彙から浮いていない。
- シーン形式が3種類以上ある。
- 例え話または小ネタが動画全体で2回以上ある。
- 想定尺に対する本編シーン数、本編セリフ数、平均セリフ数が基準を満たしている。
- 5分前後では10シーン以上、90セリフ以上、1シーン平均8セリフ以上。
- 各シーンに最低1回、追加の深掘り往復がある。
- ZMでは、ずんだもんの「なのだ」「のだ」率が20〜40%目安。10%未満または50%以上は修正対象。
- 同じ語尾が3連続していない。
- RMでは魔理沙の「だぜ」が20%未満または70%以上になっていない。
- セリフの自然文チェックで誤字っぽい表現が残っていない。
- 最終行動が「確認＋選択＋保存/整理/予定化」など、2アクション以上で具体化されている。
- 1シーン1役割になっている。
- テンプレート枠の使い方が明記されている。
- サブ枠の有無が反映されている。
- `dialogue[].text` が25文字以内。
- 全本編シーンに `visual_asset_plan` がある。
- `imagegen_prompt` が具体的。
- `imagegen_prompt` が固定フォーマットの8項目を満たしている。
- `imagegen_prompt` に scene_id、slot、主題、構図、テンプレート枠、色、余白、禁止要素がある。
- main/sub の役割が分かれ、字幕帯とキャラ位置を避ける指示がある。
- 参照動画型では参照メタが入っている。
- `meta.layout_template` が `Scene01` から `Scene21`。
- `scenes[].scene_template` を使っていない。
