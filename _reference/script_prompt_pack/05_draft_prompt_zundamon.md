# 05_draft_prompt_zundamon

## 目的

ZMペア、ずんだもん・四国めたんの高品質な解説台本を作る。

## キャラ

- ずんだもん: 視聴者代表。賢くしすぎない。勘違い、ボケ、雑な理解、焦り、ツッコミを担当。
- 四国めたん: 解説役。冷静に刺す。短く訂正し、具体例、数字、結論を出す。

## 生成前ブレーキ

台本本文を書く前に、必ず「視聴者体験設計」を出す。失敗条件は `00_MASTER_SCRIPT_RULES.md` の「視聴者体験設計」に従う。

## 設計メモの隔離

設計メモはキャラにそのまま読ませない。`script_final.md` に設計キー、監査ラベル、未充填プレースホルダを残さない。詳しい共通禁止は `00_MASTER_SCRIPT_RULES.md` を正本にする。

- 設計メモや監査項目をセリフ本文へ回収する。
- 「でも少し見るだけなら平気な気がするのだ」「小さくても入口が減れば戻りやすくなりますわ」「そこが入口の罠ですわ」をsceneごとに繰り返す。
- 「たしかに、◯◯する場面だとつい流されるね」のような、文法だけ通って状況として不自然な受けを量産する。
- 比喩を置くだけで意味を回収しない。
- ずんだもんの全発話が「のだ／なのだ」だけで構成されている（「だね」「かな」「じゃん」「マジで？」「かぁ」「でしょ？」を最低3種類混ぜる）。
- めたんの全発話末尾が「ですわ」固定（「ですの／なのです／ということですわ／ますわよ／かしら／〜よ／〜ね」を混ぜる）。

## 四国めたん 口調サンプル (必ず rotate する)

めたんの語尾は1パターンに固定しない。1本につき5語尾以上を実際に使い、同一語尾を3シーン連続で使わない。
`local_canonical/zundamon_master.md` の口調定義が正本。下記は抜粋と拡張例。

例:
- 「待ちなさい、ずんだもん。それは少し違いますわ」
- 「簡単に言うと、〇〇ということですわ」
- 「ここが今回の重要ポイントですの」
- 「実はそこに落とし穴があるのです」
- 「ずんだもん、いいところに気づきましたわ」
- 「つまり、〇〇を知らないと損をするということですわ」
- 「それ、典型ですわよ」
- 「現実はもう少し地味ですの」
- 「半分正解、半分外れかしら」
- 「気持ちはわかりますけれど」
- 「そこ、見落としがちなのです」

語尾の系統（各系統を均等に使う）:
- `〜ですわ / 〜ますわ / 〜ますわよ`: 落ち着いた断言・確信
- `〜ですの / 〜ですの？`: やや問いかけ・確認
- `〜なのです / 〜のです`: 丁寧な説明・強調
- `〜ということですわ`: まとめ・結論
- `〜かしら`: 少し呆れ・疑問
- `〜よ / 〜ね`: 親しみある口調（短文に使う）

## ずんだもんの受け方5系統テンプレ (1本につき最低4系統を実際に使う)

受け方は5系統をローテーションする。同一系統が4回連続で出てきたらFAIL。
「のだ」付与は5系統合計の40%以下に抑え、「じゃん／かな／だね／マジで？／かぁ」を混ぜる。

- **ボケ**: 「もう詰みじゃん」「コーヒー5杯でクリアなのだ」「全部オフにすれば勝ちなのだ」
- **驚き (L3)**: 「マジで！？」「え、年6万！？」「待って、それヤバいのだ」
- **疑い**: 「ほんとに？」「なんか怪しいのだ」「絶対裏あるじゃん」
- **生活実感**: 「あー、ぼくも夜それやってる」「毎月引かれてた気がするのだ」「ウチも完全にそれ」
- **行動宣言**: 「今夜やるのだ」「明日の朝にチェック」「今すぐ設定見るのだ」

「つまり、〇〇ってこと？」「つまり、〇〇ってことなのだ？」型の理解確認は**1本につき最大2回**。3回目以降はFAIL、上記5系統のいずれかに振り替える。

## 視聴者代表ボケ × 会話入り口 早見表 (ZM)

各シーンの `conversation_entry` を自然なずんだもんの1発目に変換する型例（参考）:

| conversation_entry | ずんだもんの1発目の型例 |
|---|---|
| 誤解訂正型 | 「ぼく、〇〇すれば最強だと思ってたのだ」 |
| 失敗エピソード型 | 「昨日それやって詰んだのだ」 |
| Before-After型 | 「やる前と後で何も変わらない気がするじゃん」 |
| 手順型 | 「手順？1個ずつやってたら一生終わらないのだ」 |
| 反証型 | 「みんなそう言ってるから本物なのだ」 |
| あるある型 | 「夜になるとつい〇〇しちゃうのだ」 |
| まとめ再フック型 | 「で、結局一番大事なのは何なのだ？」 |

## ずんだもんの冒頭発話ルール

各シーンのずんだもんの1発目は、原則として質問禁止。質問で始めていいのは全体の2シーンまで。

使える型:

- 勘違い: 「リンクが来たなら押すのが普通なのだ」
- 言い訳: 「忙しい時は確認なんて無理なのだ」
- 雑な解決策: 「全部オフにすれば勝ちなのだ」
- 自分に都合のいい解釈: 「上に出た検索結果なら本物なのだ」
- 怖がり: 「もう入力したら人生終了なのだ？」
- ボケ: 「スマホ、ぼくより働き者なのだ」

## 必須

- 冒頭の最初の1〜3発話は日常前提から入る。
- 「強いパスワードを使い回せば楽」のような、前提なしの雑な解決策始まりは禁止。
- 冒頭5秒で損失、衝撃数字、誤解訂正、強い疑問、あるあるからの意外性のいずれかを刺す。
- 冒頭1番目のセリフは次の5タイプから最低1つを採用する。
  - 損失提示: 「無料のつもりが、毎月980円抜かれてたのだ」
  - 衝撃数字: 「その充電、バッテリー寿命を2年縮めてるかも」
  - 誤解訂正: 「サブスク解約、アプリ消すだけじゃ止まらないのだ」
  - 強い疑問: 「なんで使ってないアプリに毎月払ってるのだ？」
  - あるある→意外: 「節約あるある。でも一番漏れてるの、買い物じゃなかったのだ」
- 「あるある」だけ、「今回は〜について」、「こんにちは」、「〇〇って知ってる？」だけの冒頭はFAIL。
- 日常の状況、視聴者がやりがちな行動、なぜ今この話をするかを置いてから、損失や危険性へ接続する。
- ずんだもんを質問係だけにしない。
- ずんだもんの受けは「驚き」「疑い」「ボケ」「生活実感」「行動宣言」の5系統から1本につき最低4系統を実際に使う。「つまり、〇〇ってこと？」「つまり、〇〇ってことなのだ？」型は1本につき最大2回（3回目以降はFAIL）。
- 1シーン内のずんだもん発話のうち、疑問形（？で終わる発話）は最大1個。それ以外は、ボケ、誤解、強い反応、ツッコミ、行動宣言、実体験のいずれかにする。
- 各本編シーンのずんだもんの1発目は、単なる質問ではなく、勘違い、言い訳、視聴者の本音、雑な解決策、焦り、疑い、的外れなたとえ、自分に都合のいい解釈のどれかにする。
- 各シーンのずんだもんの1発目を質問にしてよいのは全体で最大2シーンまで。
- ずんだもんの「なのだ」「のだ」は自然に混ぜる。全発話には付けない。
- 「だね」「かな」「じゃん」「マジで？」なども混ぜる。
- 「するのだ」「やるのだ」「行くのだ」「あるあるなのだ」「危険なのだ」「便利なのだ」「重要なのだ」「大事なのだ」のような形容詞・名詞＋「のだ」の機械結合は禁止。動詞は普通に活用させ、感情語や状況描写に「のだ」を付ける場合のみ自然なリズムを保つ。
- めたんは説明に入る前に、8〜18字程度の短いツッコミまたは訂正を入れる。
- めたんは原則2セリフ以内でずんだもんに返す。根拠説明、手順説明、注意点整理、比較説明などの深掘り時は最大4セリフまで許可する。
- 深掘りの連続解説は、前にずんだもんの誤解/疑問/ボケ/焦りがあり、説明内に具体例・数字・比較・手順・失敗例のうち最低2つがあり、後ろでずんだもんが反応/言い換え/ツッコミ/行動宣言で受ける場合だけ使う。
- 5セリフ以上続けたい場合は、途中でずんだもんの短い反応を挟む。
- L3以上リアクションを最低2回。
- 各シーンに数字、具体例、失敗例、あるある、比較のいずれかを入れる。
- 中盤シーン（全体の40〜60%地点）には、衝撃数字の再提示、失敗例の生々しい描写、視聴者煽り、比較のいずれかを最低1つ入れる。
- 最後から2番目のシーンには、投票形式の問いかけ、共感誘発のあるある、強めの言い切りのいずれかを入れる。
- 最後は具体行動で終える。
- 設計メモをメタ情報として `script_final.md` に残さない。台本品質は `audits/script_final_review.md` のLLMレビューで判断する。
- sub枠ありテンプレートでは、全シーンに会話とは別役割の `sub_content` 材料を書く。細かい変換条件は `docs/pipeline_contract.md` を正本にする。

## セリフ

- 表示都合で先に切り詰めない。
- 長い説明は、意味や掛け合いの切れ目で自然に言い換える。
- ずんだもん40、めたん60目安。
- 「なのだ」「のだ」は20〜40%目安。

## 最終自己確認

draft 出力後は、追加ファイルを読まずにこのファイルと `00_MASTER_SCRIPT_RULES.md` の範囲で確認する。

- ずんだもんが「そうなのだ」だけで受けていない。
- ずんだもんの全発話に「のだ／なのだ」が付いていない。
- めたんの語尾が「ですわ」だけに固定されていない。
- めたんが条件なしの抽象独演になっていない。
- めたんの連続解説がある場合、最大4セリフ以内で、具体素材2つ以上と前後のずんだもんリアクションがある。
- 冒頭が「今回は〇〇について解説します」になっていない。
- 各sceneに勘違い、短いツッコミ、具体例、反応、小結論がある。
- 同じ入り方、同じ返し、同じ小結論が3scene以上続いていない。

`examples/` 配下は人間監査や追加学習用の補助資料であり、通常生成の必読入力ではない。
